# AI Agent 项目工作入口

本文件是所有 AI Agent 进入本仓库后的第一阅读入口。Agent 必须先基于代码事实工作，再通过文档推进任务。代码实现与文档规划必须分离；除非用户明确要求实施某项任务，否则维护规划时不得修改业务代码。

## 1. 项目速览

- 项目：Navidrome Statistic，异步 Python 服务，为 Navidrome/Subsonic 播放状态提供轮询、播放计数和统计页面。
- 运行时：Python 3.11（`Dockerfile` 当前事实）、FastAPI、httpx、aiosqlite、SQLite。
- 主入口：`src/main.py`；HTTP 客户端：`src/client.py`；数据库：`src/database.py`；页面：`src/static/index.html`。
- 默认服务端口：`39421`。
- 测试：`tests/`，执行入口为 `pytest`。
- 当前实现边界与已知差异：见 [`docs/current-state.md`](docs/current-state.md)。

## 3. 每次工作前

1. 运行 `git status --short --branch`，识别用户已有修改。不得覆盖、回滚或格式化无关修改。
2. 阅读本文件、`docs/README.md`、`docs/current-state.md`、`docs/interfaces.md`、`docs/privacy.md` 和 `docs/tasks.md`。
3. 阅读目标任务涉及的源码、配置和测试；不得仅根据 README 或任务描述推断实现事实。
4. 在 `docs/tasks.md` 通过唯一 ID 领取一个可执行任务，检查其 `状态`、`依赖`、`涉及文件`、隐私确认和验收标准。
5. 涉及个人信息、服务器地址、用户名、密码、token、数据库真实内容或日志样本时，先执行隐私检查。标为“需用户人工确认/编辑”的内容必须暂停并请求用户处理，AI 不得猜测或代填。
6. 记录基线验证命令及结果。若依赖缺失导致无法运行，应记录“环境阻塞”，不得声称测试通过。

## 4. 工作进行中

1. 将任务状态按 `待办 -> 进行中 -> 阻塞/待验收 -> 已完成` 更新；同一时间避免领取多个互不相关的任务。
2. 实现只覆盖任务边界。规划、风险和未来工作写入 `docs/tasks.md`，不要用 TODO 注释替代任务管理。
3. 任何 HTTP API、Subsonic 请求/响应字段、环境变量或数据库 schema 变化，都必须同步更新 `docs/interfaces.md`，说明稳定性、兼容策略和验证方式。
4. 不在文档、测试 fixture、日志、命令、提交信息中写入真实凭据。只可记录变量名和来源，例如 `NAVIDROME_PASS` 来源为部署环境或本地未跟踪 `.env`。
5. 接口不确定时标为“待确认”或“实验性”，附代码位置与验证步骤，不得将推测写为稳定事实。
6. 若发现新工作，新增具有唯一 ID 的任务并补齐规定字段；不要直接扩大当前任务范围。

## 5. 每次工作后

1. 运行任务中的验证命令，以及与改动范围匹配的测试、lint、容器配置检查。
2. 检查 `git diff --check`、`git diff --stat`、`git diff` 和 `git status --short`；确认未覆盖用户无关修改，未纳入 `.env`、数据库或真实敏感值。
3. 检查文档相对链接和接口登记是否同步。文档内命令必须能从仓库根目录执行，或明确工作目录。
4. 更新任务状态和“完成记录”：日期、实施者、变更摘要、验证结果、提交/PR（如有）、遗留风险。未满足全部验收标准不得标记“已完成”。
5. 若接口发生变化，执行 `docs/interfaces.md` 的变更流程并记录兼容性结论。
6. 向用户报告实际完成内容、验证结果、阻塞项和需人工确认的隐私项，不回显敏感值。

## 6. 任务维护硬性规则

- 唯一来源：后续推进任务只写入 `docs/tasks.md`；其他文档可链接任务 ID，但不维护平行清单。
- ID 永不复用或重编号。建议格式：`NDS-<领域>-<三位序号>`。
- 每项必须包含：优先级、状态、依赖、目标、实施步骤、验收标准、验证命令、涉及文件、风险/回滚、完成记录。
- `已完成` 任务保留，不删除；取消任务标记 `已取消` 并说明原因和替代项。
- 任务范围变化时更新原任务并记录原因；形成独立交付物时新建任务并建立依赖。
- 发现代码事实与文档不一致时，先更新当前事实或登记差异，再决定是否创建实施任务。

## 7. 接口与隐私硬性规则

- 接口登记的四类范围：本服务 API、上游 Subsonic API、环境变量、SQLite schema。
- 稳定性等级统一使用 `稳定`、`受支持但可演进`、`实验性`、`内部`、`待确认`。
- 破坏性变更必须先建立任务，给出迁移/兼容窗口、回滚方案和验证证据；数据库变更必须考虑既有数据。
- 敏感值只记录变量名、责任方和来源，不记录值。示例值必须明显为占位符。
- `username`、播放历史、曲名、艺人、专辑、客户端名称可构成个人行为数据，默认按隐私数据处理。
- 任何真实部署地址、公开访问策略、数据保留周期、备份位置与授权边界均为“需用户人工确认/编辑”。
